Este documento apresenta a especificação dos Requisitos Não-Funcionais (RNFs) e a fundamentação do Padrão Arquitetural adotado para o desenvolvimento do sistema SafePlace. O objetivo deste artefato é garantir que a estrutura do software alinhe-se rigorosamente às necessidades de desempenho, segurança, conformidade legal e escalabilidade exigidas pelo projeto.

---

## 1. Requisitos Não-Funcionais (RNFs)

Os requisitos a seguir delimitam os critérios de qualidade, restrições técnicas e níveis de serviço que o ecossistema SafePlace deve cumprir.

| Código | Requisito                                  | Descrição                                                                                                                                                                                                                                                                                                                                                                                        |
| ------ | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| RNF01  | Tempo de resposta                          | O sistema deve responder às interações do usuário em até 3 segundos em condições normais de uso, e em até 1 segundo para consultas simples de estoque e cadastros.                                                                                                                                                                                                                               |
| RNF02  | Monitoramento em tempo real                | O sistema deve processar e exibir dados de sensores ambientais com latência máxima de 500 milissegundos, garantindo respostas imediatas em situações de emergência.                                                                                                                                                                                                                               |
| RNF03  | Capacidade de usuários simultâneos         | O sistema deve suportar ao menos 200 usuários simultâneos sem degradação perceptível de desempenho, sendo escalável conforme o crescimento da organização.                                                                                                                                                                                                                                        |
| RNF04  | Controle de acesso por perfil              | O sistema deve implementar controle de acesso baseado em perfis (RBAC), restringindo funcionalidades conforme o papel do usuário, sendo eles: operador, gestor de segurança e administrador.                                                                                                                                                                                                     |
| RNF05  | Criptografia de dados sensíveis            | Todos os dados sensíveis, como atestados de saúde, CATs, históricos de acidentes e mídias anexadas, devem ser armazenados e transmitidos com criptografia AES-256 e protocolo HTTPS com TLS 1.3.                                                                                                                                                                                                 |
| RNF06  | Rastreabilidade de ações (auditoria)       | O sistema deve manter log imutável de todas as operações de criação, edição e exclusão de registros, com identificação do usuário responsável, data, hora e dado alterado, armazenado por no mínimo 5 anos.                                                                                                                                                                                      |
| RNF07  | Disponibilidade mínima                     | O sistema deve garantir disponibilidade de 99,5% (uptime), com janela de manutenção planejada de no máximo 4 horas mensais, preferencialmente fora do horário de expediente.                                                                                                                                                                                                                     |
| RNF08  | Backup e recuperação de dados              | O sistema deve realizar backup automático diário dos dados, com capacidade de recuperação (RPO) de até 24 horas e tempo de restauração (RTO) de até 4 horas em caso de falha crítica.                                                                                                                                                                                                            |
| RNF09  | Funcionamento offline parcial              | As funcionalidades críticas de registro de acidentes e incidentes devem operar em modo offline por meio de recursos de Progressive Web App (PWA), armazenando os dados localmente no dispositivo e sincronizando automaticamente com o servidor quando a conexão for restabelecida. Funcionalidades que dependem de processamento em tempo real ficam indisponíveis durante a ausência de conexão. |
| RNF10  | Conformidade com normas regulamentadoras   | O sistema deve ser aderente às Normas Regulamentadoras do MTE aplicáveis, especialmente NR-1, NR-6 e NR-9, à LGPD (Lei 13.709/2018) para dados pessoais de saúde, e à legislação previdenciária aplicável à emissão de CAT.                                                                                                                                                                      |
| RNF11  | Integridade dos documentos gerados         | Os documentos gerados pelo sistema, como relatórios, CATs e laudos, devem conter assinatura digital ou hash de integridade para garantir autenticidade perante órgãos fiscalizadores.                                                                                                                                                                                                             |
| RNF12  | Interface responsiva e acessível           | A interface do sistema deve ser responsiva, funcionando adequadamente em dispositivos desktop, tablet e mobile, e deve seguir as diretrizes de acessibilidade WCAG 2.1 nível AA, permitindo uso por pessoas com deficiências visuais ou motoras.                                                                                                                                                  |
| RNF13  | Facilidade de aprendizado                  | Um usuário com conhecimento básico de informática deve ser capaz de realizar as tarefas principais do sistema após no máximo 2 horas de treinamento.                                                                                                                                                                                                                                             |
| RNF14  | Suporte multilíngue                        | O sistema deve oferecer interface em português do Brasil como idioma padrão, com possibilidade de expansão para espanhol e inglês sem necessidade de refatoração do código-fonte.                                                                                                                                                                                                                 |
| RNF15  | Modularidade e extensibilidade             | O sistema deve ser desenvolvido em arquitetura modular, permitindo a adição de novos módulos sem impacto nas demais funcionalidades já existentes.                                                                                                                                                                                                                                                |
| RNF16  | Documentação técnica                       | O sistema deve possuir documentação técnica atualizada, contemplando API, modelo de dados e arquitetura, de forma que um desenvolvedor externo consiga compreender e realizar manutenções sem suporte da equipe original em até 5 dias úteis.                                                                                                                                                      |
| RNF17  | Compatibilidade com navegadores e sistemas | O sistema web deve ser compatível com as duas versões mais recentes do Chrome, Firefox, Edge e Safari, e deve poder ser implantado em ambientes com Linux e Windows Server.                                                                                                                                                                                                                       |
| RNF18  | Integração via API                         | O sistema deve expor uma API RESTful documentada no padrão OpenAPI 3.0 para integração com sistemas externos, como ERPs, sistemas de RH e plataformas de sensores IoT, utilizando autenticação OAuth 2.0.                                                                                                                                                                                        |

## 2. Padrão Arquitetural Adotado: Arquitetura Híbrida

O sistema SafePlace adota uma **Arquitetura Híbrida** baseada em Camadas Lógicas organizadas de forma descendente, com a incorporação de Fronteiras Hexagonais (Ports & Adapters) para isolar o domínio de negócios, e um Barramento de Mensageria Orientado a Eventos (Event-Driven Architecture — EDA) para o processamento em tempo real.

### 2.1. Por que arquitetura em camadas como base?

A esmagadora maioria dos requisitos (RF03 a RF07, RF09, RF11, RF12, RF14 a RF23) é essencialmente CRUD com regras de negócio — gestão de estoque, treinamentos, documentos, relatórios, CAT, validade de ASOs. Esses requisitos se encaixam perfeitamente em camadas bem separadas (apresentação → aplicação → domínio → infraestrutura), que é o padrão mais didático, manutenível e adequado para equipes acadêmicas e empresas de médio porte. Atende diretamente ao RNF15 (modularidade) e RNF16 (documentação/manutenibilidade).

### 2.2. Justificativa da Escolha da Arquitetura Híbrida

A fusão de conceitos de Camadas, Hexagonal e Eventos foi escolhida de forma estratégica para balancear a simplicidade de desenvolvimento de uma equipe de médio porte/acadêmica com o rigor tecnológico exigido pelos requisitos de negócios:

- **Camadas Lógicas (Macro-organização):** A esmagadora maioria dos requisitos do SafePlace refere-se a operações transacionais tradicionais (CRUD) com regras de negócio para gestão de estoque, treinamentos, documentos, laudos e emissão de CAT. A divisão macro em `apresentacao`, `aplicacao`, `dominio` e `infraestrutura` provê uma excelente separação de conceitos, acelerando o desenvolvimento e facilitando a manutenibilidade (RNF15 e RNF16).

- **Inversão de Dependência e Fronteiras Hexagonais (Isolamento de Negócio e LGPD):** Para garantir o cumprimento do RNF10 (LGPD e Normas Regulamentadoras) e do RNF15 (Extensibilidade), o domínio não pode depender de detalhes de infraestrutura. Por isso, isolamos o domínio por meio de interfaces de repositórios (Portas de saída, ex: `IRepositorioDeSensor`, `IRepositorioDeAcidente`). A infraestrutura (Adaptadores de persistência) implementa esses contratos. As comunicações com sistemas legados e telemetria são feitas de forma desacoplada por meio de adaptadores específicos (`AdaptadorDeSensorIOT`, `AdaptadorERP`, `AdaptadorESocial`).

- **Barramento de Eventos (Tempo Real e Desempenho):** O monitoramento ambiental em tempo real (RNF02 — latência máxima de 500 ms) e os alertas automáticos reativos não podem rodar em caminhos de requisição/resposta síncronos convencionais. O módulo de mensageria (infraestrutura) consome sinais assíncronos que são disparados para o `dominio.eventos` (ex: `SensorCriticoDetetado`, `EmergenciaAtivada`). Os manipuladores correspondentes na camada de aplicação processam essas mensagens sem onerar ou travar as operações transacionais do sistema.

## 3. Comparativo de Padrões Arquiteturais

A tabela abaixo detalha e justifica por que a combinação híbrida adotada é superior aos padrões puristas concorrentes para o cenário do SafePlace.

| Padrão Arquitetural                   | Descrição                                                                                                                            | Por que não é o melhor caminho isolado para o SafePlace?                                                                                                                                                                                                                                                                                                 |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Camadas Puro (Pure Layered)**       | Organiza o sistema em níveis horizontais estritos (Apresentação, Negócio, Persistência), onde o fluxo e as dependências são lineares. | Cria acoplamento direto entre as regras de negócio e a tecnologia de banco de dados, violando o RNF15. Não oferece suporte para processamento reativo assíncrono; forçar telemetria IoT em canais síncronos gera gargalos e estoura o limite de 500 ms (RNF02).                                                                                            |
| **Microsserviços (Microservices)**    | Divisão do sistema em dezenas de serviços independentes, cada um com sua base de dados e ciclo de deploy.                            | Introduz complexidade operacional desproporcional ao porte do sistema (Kubernetes, service mesh, transações distribuídas). O sistema exige consistência transacional rápida entre acidentes, EPIs e funcionários, o que tornaria o microsserviço inviável para 200 usuários simultâneos (RNF03).                                                            |
| **Monolítico Puro**                   | Aplicação construída em bloco único sem separação formal de camadas, com lógica de negócio misturada à interface.                    | Não atende à modularidade exigida pelo RNF15 nem à latência de tempo real para sensores (RNF02). Misturar lógica transacional e processamento contínuo de sensores em uma única estrutura inviabilizaria a escalabilidade seletiva e a segurança.                                                                                                         |
| **Event-Driven Puro**                 | Toda a comunicação ocorre por mensagens assíncronas, sem chamadas síncronas entre os módulos.                                        | Adequado para módulos reativos (sensores, alertas), mas inviável como arquitetura base única. Operações transacionais cruciais como emissão de CAT (RNF11), controle de estoques e de ASOs exigem feedback síncrono imediato para o usuário.                                                                                                              |
| **Hexagonal Puro (Ports & Adapters)** | Domínio isolado do mundo externo por portas (interfaces) e adaptadores (implementações).                                             | Utilizado parcialmente como pilar da nossa solução híbrida. Adotá-lo de forma purista em 100% dos fluxos CRUD simples adicionaria um overhead desnecessário de mapeamento de objetos para uma equipe de médio porte/acadêmica. A solução híbrida usa o Hexagonal estritamente nas fronteiras onde há volatilidade técnica (sensores, integrações ERP/eSocial, persistência). |
| **Clean Architecture**                | Organiza o sistema em anéis concêntricos impondo a regra rígida de dependências de fora para dentro.                                | Muito conceitual e robusta, mas exige um grande volume de código boilerplate (interactors, presenters, boundaries) que eleva drasticamente a curva de aprendizado da equipe em termos acadêmicos, sem entregar valor prático superior à arquitetura híbrida proposta.                                                                                      |
| **MVC (Model-View-Controller)**       | Separação clássica em dados (Model), telas (View) e fluxos de controle (Controller).                                                | É um padrão de apresentação, não uma arquitetura corporativa completa. Não dita como tratar persistência, auditoria de 5 anos (RNF06), regras regulamentadoras complexas (RNF10) ou sensores IoT em tempo real (RNF02). O MVC é incorporado dentro do pacote `apresentacao` como padrão de design de interface.                                           |
